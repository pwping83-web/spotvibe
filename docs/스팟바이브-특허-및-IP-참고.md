# 스팟바이브 — 특허·지식재산 참고 메모

> **법률 자문이 아닙니다.** 출원·등록 가능 여부는 변리사·변호사와 사건 단위로 검토해야 합니다.  
> 아래는 **한국 특허 실무에서 자주 쓰는 판단 축**과, **현재 코드베이스 수준**을 대입한 **내부용 체크리스트**입니다.

---

## 1. 특허로 “보호”되는 것의 성격

특허는 주로 **기술적 해결 수단(발명)** 에 대해, 신규성·진보성 등을 만족할 때 부여됩니다.

- **UI/UX만**인 경우: 화면 배치 자체만으로는 출원이 어렵거나, **디자인등록(의장)** ·**저작권(코드/그래픽)** 쪽이 더 맞는 경우가 많습니다.  
- **알고리즘·데이터 처리·시스템 구성**이 구체적이고, 선행 기술 대비 **비자명한 해결**을 보이면 특허 검토 가치가 올라갑니다.

---

## 2. 한국에서 자주 보는 3가지 (아주 요약)

| 요건 | 의미(비법률적 요약) |
|------|---------------------|
| 산업상 이용 가능성 | 실제 제품·서비스에 쓸 수 있는 형태 |
| 신규성 | 출원 전에 세계 어디에도 동일한 발명이 없어야 함 |
| 진보성(또는 특허법상 진보성에 해당하는 판단) | 통상의 기술자가 선행 기술만으로 쉽게 떠올리기 어려운 정도의 차이 |

---

## 3. 현재 SpotVibe 코드와 대입했을 때

### 3.1 “그대로는” 특허 대상이 되기 어려운 부분

- **Leaflet + OSM/Carto 타일 + Geolocation**: 업계 관행·공개 라이브러리 조합에 가깝습니다.  
- **탭 네비·BottomSheet·다크 테마 지도**: 일반적인 모바일 앱 패턴입니다.  
- **더미 이벤트/알림 데이터**: 기술 발명이라기보다 콘텐츠·기획 시연입니다.  
- **프로필 기반 “AI” 추천 순서·카운트다운·말풍선 인도·심장 박동 오버레이** 등: 클라이언트만의 **휴리스틱·하드코딩 규칙·단순 애니메이션** 수준이면, **특정 기술 과제를 해결하는 구체 수단**으로 보기 어렵고 UI·기획·저작권에 가깝습니다. **화면에 구현되어 있다고 해서 자동으로 특허 대상이 되지는 않습니다.**  
- **지역 기반 이벤트 탭·파트너 슬롯·수익 모델**(과금 단위 등): **사업·마케팅 구조**만으로는 특허 발명으로 다루기 어려운 경우가 많고, **표시광고법·플랫폼 정책**은 별도 준수 이슈입니다. 다만 **입찰·노출 순위·개인화 노출**을 **서버·알고리즘으로 구체화**한 부분이 생기면, 그 **기술 블록**만은 별도 발명 축으로 검토할 수 있습니다(선행조사·진보성 판단 필요).

→ 위만으로는 **진보성 주장이 약해지기 쉽습니다.**

### 3.2 “발전시키면” 검토할 만한 아이디어 축 (예시)

아래는 **아직 구현되지 않았거나 일부만 있는 개념**을, 출원용으로 구체화할 때 자주 쓰는 방향입니다.  
**실제 출원 가능 여부는 선행조사 후 판단**해야 합니다.

1. **다원 데이터 융합**  
   실시간 인구·교통·이벤트 티켓·PM 가용·날씨 등을 **가중·시간창**으로 합성해 “핫스팟 점수”를 산출하는 **수학·파이프라인**을 문서·도면으로 고정.

2. **개인 이동 프로필 기반 경로·시간 추천**  
   `MobilityProfile` 같은 사용자 속성과, **혼잡도·회피 구간**을 입력으로 받아 **ETA·행동 카드**를 생성하는 규칙/ML 파이프라인을 **구체 알고리즘**으로 기술.

3. **프라이버시 보존 위치 집계**  
   원시 좌표를 서버에 모으지 않고 **집계·노이즈**만 올리는 방식 등, **기술적 효과(정확도 vs 익명성)** 가 명확한 경우.

4. **줌·LOD와 클러스터의 연동**  
   코드에 줌 추적 훅이 있으므로, **특정 줌 구간에서만 표시하는 데이터 레이어 전환 규칙**을 독자적으로 설계했다면 그 **규칙+효과**를 발명으로 정리할 여지는 있음(선행 지도 앱과 비교 필수).

## 4. 특허 말고 챙길 IP

| 유형 | 이 프로젝트에서 해당될 수 있는 것 |
|------|-----------------------------------|
| 저작권 | 소스 코드, UI 문구, 독자 그래픽·일러스트 |
| 상표 | “스팟바이브”, “SpotVibe” 등 서비스명·로고(출원 전 동일/유사 상표 조회 권장) |
| 디자인등록 | 앱 아이콘, 독특한 UI 형태(기능과 분리된 외관) |
| 영업비밀 | 실제 운영 시 사용하는 추천 모델 가중치, 제휴 데이터 처리 방식 등 비공개 정보 |

## 5. 다음 액션(실무 체크리스트)

1. **선행기술조사**: “핫스팟·실시간 이벤트 지도”뿐 아니라 **위치 기반 추천(LBS)·LBS 광고·지도 군집·클러스터·LOD(줌 단계별 표시)·개인화 이동 안내·모빌리티 프로필** 등 키워드로 특허·논문을 넓게 검색하고, 차별점·회피 설계에 반영합니다.  
2. **발명의 포인트 한 문장**: “무엇을 **어떤 입력**으로 **어떻게** 해결하는가”.  
3. **도면·실시예**: 블록도, 시퀀스, 가짜 데이터로도 동작하는 **구체 예**.  
4. **변리사 상담**: 출원 전략(특실용·디자인 병행 등).
