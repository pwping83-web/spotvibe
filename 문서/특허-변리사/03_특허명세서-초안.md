### 1.2 서비스 정의

스팟바이브(SpotVibe)는 **사용자의 실시간 위치 및 개인 성향 데이터를 기반**으로, "지금 이 근처에서 무엇이 일어나고 있는지"를 지도 위에 시각화하고, AI 방식의 행동 지시(이동 수단 선택·시간대 조언·무드 추천)를 제공하는 **하이퍼로컬 실시간 활기(Vibe) 안내 플랫폼**이다.

핵심 가치는 다음 세 가지로 요약된다.

1. **핫스팟(Hotspot) + 바이브(Vibe) 시각화**: 특정 구역(예: 홍대·연남 일대, 강남 등)에서 연령대별 군집(클러스터), 현장 이벤트, 개인 성향 교집합 구역을 다층 오버레이로 시각화한다.

2. **개인화된 탐색 중심(Explore Anchor) 시스템**: 사용자가 관심 거점을 자유롭게 전환하면, 해당 좌표를 중심으로 인구 분포·이벤트 핀·AI 추천 앵커가 동기화되어 "내가 보는 곳"과 데이터가 일치하게 작동한다.

3. **이동 수단 프로필 기반 행동 지시형 AI UI**: 사용자가 설정한 이동 수단(자차·킥보드·뚜벅이/따릉이)에 따라 경로 안내 문구, 예상 소요 시간, 이동 팁, 바이브 팁 등 맥락에 맞는 카드형 행동 지시를 자동으로 생성하여 제공한다.

4. **AI·메타데이터·집단신고 연동 UGC 거버넌스(공익·신뢰)**: 하이퍼로컬 **실시간 현장 제보·SOS** 등 사용자 생성 콘텐츠가 **무분별한 상업 홍보·성인·범죄 악용·허위 사진**으로 오염되는 것을 막기 위해, **텍스트 LLM 판정**, **비전 LLM 판정**, **촬영 시각(EXIF) 등 클라이언트 무결성 검사**, **유형별 신고·서로 다른 신고자 수 기반 자동 제한**, **관리자 재판·해제**를 **단일 파이프라인**으로 결합한다. 이는 단순 광고 플랫폼이 아니라 **이웃 안전·정보 신뢰**를 우선하는 **공익적 운영 목적**과 정합된다.

### 1.3 목표 사용자

- 라이브·버스킹·야외 모임 등 **"지금 어디가 붐비는지"** 를 빠르게 파악하고 싶은 20~40대 모바일 사용자
- 대중교통·개인형 이동수단(PM)·자차 등 **이동 방식이 서로 다른 사용자** 각각에게 최적화된 한 줄 이동 안내를 원하는 사용자
- 지역 행사·공연·버스킹 정보를 **지도 기반으로** 탐색하고 싶은 하이퍼로컬 서비스 수요 사용자

#### [STEP 3] 지도 탭 진입 및 개인화 오버레이 생성

```
사용자: [지도] 탭 탭
        │
        ▼
  MapArea.tsx 마운트
        │
  ┌─────┴──────────────────────────────────────────────┐
  │  (A) 위치 취득 레이어                               │
  │  navigator.geolocation.watchPosition()              │
  │  → 1,200ms 쓰로틀링 적용 후 myLatLng 상태 갱신     │
  │  → Leaflet flyTo(center, zoom, {duration:0.85})     │
  │    (첫 GPS 고정 시 1회만 지도 이동, 이후는 마커만)  │
  └─────────────────────────────────────────────────────┘
        │
  ┌─────┴──────────────────────────────────────────────┐
  │  (B) 연령대별 가상 군중 분포 생성(PRNG 알고리즘)    │
  │                                                     │
  │  driftSeed = floor(Date.now() / 60_000)             │
  │    → 1분마다 시드 갱신 (군중 흐름 드리프트 효과)    │
  │                                                     │
  │  randomPointsInDisk(                                │
  │    centerLat, centerLng,                            │
  │    count,                                           │
  │    seed = driftSeed × ageGroupMultiplier,           │
  │    maxRadiusKm                                      │
  │  )                                                  │
  │  내부 알고리즘:                                     │
  │    rnd = mulberry32(seed)   ← 결정론적 PRNG         │
  │    for i in count:                                  │
  │      angle = rnd() × 2π                             │
  │      t     = √(rnd())      ← 균일 원판 분포 보정   │
  │      dLat  = rLat × t × cos(angle)                 │
  │      dLng  = (rLat × t × sin(angle)) / cosLat      │
  │      → [centerLat+dLat, centerLng+dLng] 저장        │
  │                                                     │
  │  생성 결과:                                         │
  │    20대 군중 → CircleMarker(color=#FFDE00, r=4px)   │
  │    30대 군중 → CircleMarker(color=#FF6B6B, r=4px)   │
  │    40대+ 군중 → CircleMarker(color=#00F0FF, r=4px)  │
  └─────────────────────────────────────────────────────┘
        │
  ┌─────┴──────────────────────────────────────────────┐
  │  (C) 개인화 오버레이 생성 (프로필 연동)              │
  │                                                     │
  │  MBTI 오버레이:                                     │
  │    mbtiSet.forEach(type =>                          │
  │      MBTI_OVERLAY[type]을 참조하여                   │
  │      exploreCenter + offset 위치에                  │
  │      Circle(color, radiusM) 렌더링                  │
  │    )                                                │
  │                                                     │
  │  활동 태그 오버레이:                                │
  │    activityTags.forEach(tag =>                      │
  │      ACTIVITY_OVERLAY[tag]을 참조하여               │
  │      Circle(color, radiusM) 렌더링                  │
  │    )                                                │
  └─────────────────────────────────────────────────────┘
        │
  ┌─────┴──────────────────────────────────────────────┐
  │  (D) 심장 박동 오버레이 애니메이션                   │
  │                                                     │
  │  hbTick: setInterval 95ms 간격으로 틱 카운터 갱신   │
```

```
        │
  ┌─────┴──────────────────────────────────────────────┐
  │  scoreAiRec(item, mbtiSet, genderPref, activityTags)│
  │                                                     │
  │  점수 산출 규칙:                                    │
  │    item.activityTags와 사용자 activityTags 교집합  │
  │      → 일치 태그 1개당 +3점                         │
```

### 3.1 특허 포인트 ① — 사용자 다중 성향 속성의 집합 연산 기반 지리적 오버레이 생성 방법

#### 기술적 과제

기존 위치 기반 서비스(LBS)는 단일 속성(예: 연령대 또는 관심사 하나)으로만 지도 레이어를 필터링하여, 복합적인 성향을 가진 사용자에게 정밀한 구역 시각화를 제공하지 못한다.

#### 해결 수단 (본 발명의 구체적 구현)

스팟바이브는 사용자 프로필을 다음과 같이 **복수의 독립 속성 집합**으로 구성한다.

- **MBTI 집합** (`mbtiSet`): 사용자가 선택한 1~16개의 MBTI 유형
- **활동 태그 집합** (`activityTags`): 운동·수다·산책·야외·소풍·공연·맛집·쇼핑·클럽·카페·전시·야경 등 12개 태그
- **성별 인파 선호** (`genderPref`): 전체/여성 다수/남성 다수

각 속성은 **독자적인 지리 오프셋(offset)과 반경(radiusM)** 을 가지고 있으며, 탐색 중심 좌표를 기준으로 해당 오프셋만큼 이동한 위치에 Circle 오버레이를 렌더링한다.

```
예시:
  MBTI = {ENFP, ESTP}이고 활동 태그 = {공연, 클럽}인 경우:
    → 지도 위에 4개의 색상 구분 오버레이 원이 동시 표시
    → 각 원의 크기와 색상이 속성 유형에 따라 독립적으로 설정됨
    → 복수 원의 시각적 교집합 영역이 "나와 맞는 구역"으로 직관적으로 표현
```

이를 통해 **"다중 성향 교집합 구역"** 을 지도 위에서 실시간으로 시각화하는 기술적 효과를 달성한다.

#### 청구항 후보 (안)

> **청구항 제1항**: 사용자의 MBTI 유형 집합, 활동 관심사 태그 집합, 및 성별 인파 선호 속성을 각각 독립된 지리 좌표 오프셋 및 반경값에 매핑하고, 탐색 중심 좌표를 기준으로 각 속성에 대응하는 복수의 반투명 원형 오버레이를 지도 위에 동시 렌더링함으로써, 사용자의 복합 성향이 집중되는 지리적 구역을 시각화하는 방법.

### 3.2 특허 포인트 ② — 결정론적 PRNG 기반 시간 동기화 군중 분포 드리프트 시스템

#### 기술적 과제

실제 실시간 인구 데이터를 서버에서 수집·가공하여 클라이언트에 전송하는 방식은 사용자 개인정보 수집 부담과 높은 서버 비용을 수반한다. 그러나 데이터가 없는 경우 군중 분포를 전혀 시각화할 수 없어 사용자 경험이 저하된다.

#### 해결 수단 (본 발명의 구체적 구현)

스팟바이브는 다음의 **결정론적 유사난수 발생(PRNG) 기반 군중 시뮬레이션** 방식을 구현한다.

**① 시드 생성 (1분 단위 시간 동기화)**

```
driftSeed = Math.floor(Date.now() / 60_000)
  → 같은 1분 구간 안에 있는 모든 사용자 인스턴스는
    동일한 시드를 생성 → 동일한 군중 분포를 확인
```

**② mulberry32 결정론적 PRNG**

```
function mulberry32(seed):
  return function():
    t = (seed += 0x6D2B79F5)
    t = imul(t ^ (t >>> 15), t | 1)
    t ^= t + imul(t ^ (t >>> 7), t | 61)
    return ((t ^ (t >>> 14)) >>> 0) / 4294967296
    → [0, 1) 범위의 결정론적 부동소수점 수열 생성
```

**③ 균일 원판 분포 좌표 샘플링**

```
for i in count:
  angle = rnd() × 2π                    (방향각 균일 분포)
  t     = √(rnd())                      (반경: 면적 보정을 위한 제곱근 변환)
  dLat  = rLat × t × cos(angle)
  dLng  = (rLat × t × sin(angle)) / cosLat  (위도에 따른 경도 보정)
  → [centerLat + dLat, centerLng + dLng]
```

이 방식은 **서버 인구 데이터 없이** 1분 주기로 자연스럽게 이동하는 군중 흐름을 클라이언트에서 완전히 재현 가능하게 시뮬레이션하며, 동일 시드를 사용하는 모든 클라이언트에서 **동기화된 일관성 있는 군중 분포** 를 보장한다.

#### 청구항 후보 (안)

> **청구항 제2항**: 현재 시각의 Unix 타임스탬프를 60,000으로 나눈 정수 값을 PRNG 시드로 사용하여, 복수의 클라이언트 인스턴스가 서버 통신 없이도 동일한 1분 구간 내 동기화된 군중 분포 좌표를 생성하고, 제곱근 변환을 적용한 균일 원판 샘플링으로 지리 좌표를 생성하여 지도 위에 연령대별 색상 구분 마커로 렌더링하는 방법.

### 3.3 특허 포인트 ③ — 이동 수단 프로필 기반 문맥 적응형 행동 지시 카드 생성 방법 (BM 특허)

#### 기술적 과제

기존 O2O·지도 서비스는 사용자의 이동 수단에 관계없이 동일한 경로 정보를 제공하거나, 이동 수단 선택을 경로 탐색 시점에서만 일시적으로 반영한다. 이로 인해 사용자는 자신의 이동 상황에 맞는 맥락적 조언(시간대 팁, 현장 무드 팁 등)을 통합적으로 받을 수 없다.

#### 해결 수단 (본 발명의 구체적 구현)

스팟바이브는 사용자가 **사전에 등록한 이동 수단 프로필**(MobilityProfile)을 앱 전역에 유지하고, 특정 지역 군집(클러스터)을 탭하는 행동 시점에 해당 프로필을 조회하여 **이동 수단별로 분기된 경로 안내 문구** 와 **3종의 행동 지시 카드** 를 자동으로 생성한다.

| 이동 프로필 | 생성되는 경로 안내 내용 |
|-------------|------------------------|
| `pedestrian_ddareungi` | 버스 노선 + 따릉이 거치대 연계 안내 |
| `car_owner` | 실시간 정체 회피 + 네이버 내비 ETA 연동 안내 |
| `kickboard_license` | 반경 내 PM 대기 대수 + 산책로 전환 안내 |

행동 지시 카드는 **이동 팁·타임라인·바이브 팁** 3종으로 고정 구성되어, 물리적 이동 조언(수단·경로)과 비물리적 조언(시간·무드)을 통합하여 제공한다.

또한, 외부 지도 서비스(네이버 지도) 연동 URL을 `naverMapSearchUrl(destination)` 함수로 동적 생성하여, 앱 내 안내와 외부 내비게이션의 연계 동선을 설계한다.

#### 청구항 후보 (안)

> **청구항 제3항**: 사용자가 사전 등록한 이동 수단 프로필을 앱 세션 전역에 유지하고, 지도 위 특정 군집 지점에 대한 사용자 인터랙션 발생 시, 해당 프로필 값에 기반하여 이동 경로 안내·최적 시간대 조언·현장 무드 팁을 포함하는 복수의 문맥 적응형 행동 지시 카드를 자동 생성하여 하단 시트 UI로 제공하는 방법 및 시스템.

### 3.4 특허 포인트 ④ — AI·촬영 메타데이터·집단 신고 연동 UGC 거버넌스 및 공익성

#### 기술적 과제

하이퍼로컬 **실시간 현장 정보·이웃 SOS** 채널은 **무분별한 상업 홍보·성인 콘텐츠·범죄 연계 문구·허위·재촬영 사진**이 유입되면 **공공의 신뢰와 안전**을 해친다. 단일 규칙 필터(키워드 차단)만으로는 은어·이미지 기반 스팸·화면 재촬영을 막기 어렵고, **무조건 통과**는 플랫폼 오남용을 부추긴다.

#### 해결 수단 (본 발명의 구체적 구현 — 다층 방어)

스팟바이브는 **같은 서비스 내에서** 다음을 **조합**한다.

| 층 | 역할 | 구현 요지 |
|----|------|-----------|
| **A. 텍스트 LLM** | 현장 제보 제목·설명, SOS 메모 | Edge `moderate-user-content`: 맥락별 `allow` / `held` / `block`(또는 SOS는 `allow`/`block`). **held** 시 공개 노출 전 **관리자·정책 큐**로 유도. |
| **B. 비전 LLM** | SOS 첨부 사진(선택) | Edge `verify-sos-photo`: 화면 재촬영·UI·명백 광고/성적 남용 등 **abuse** 판정, 요약·이유 JSON. |
| **C. 클라이언트 EXIF·스크린샷 휴리스틱** | 허위·다운로드·캡처 | **압축 전 원본**에서 촬영 시각·Software 태그 검사, **시간 창** 밖·메타 부재·스크린샷 의심 시 **AI 호출 전 차단**. |
| **D. 집단 신고·자동 제한** | 동료 검증 | 서로 다른 신고자 수가 임계에 도달하면 `user_moderation` 등으로 **활동 제한** 후행, **오판 시 관리자 복구** 경로. |
| **E. 관리자 재판** | 최종 공정성 | held 제보·신고 큐에서 **승인·반려·숨김** RPC 등으로 **인간 최종 판단**을 병행. |

**공익성**: 상기 파이프라인은 수익형 노출과 별도로, **시민 안전(SOS)·지역 정보 신뢰**를 **기술적으로 선제 방어**하는 구조이며, “허위·선정·광고 스팸이 올라가는 것”보다 **차단·보류 쪽을 우선**하는 정책을 **시스템에 내장**할 수 있다.
