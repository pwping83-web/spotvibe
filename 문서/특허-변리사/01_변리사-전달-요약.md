## 1. 발명 요지(2~3문장)

하이퍼로컬 UGC(현장 제보)·SOS 채널에 대해, **텍스트 AI 판정(허용·보류·차단)** 과 **보류 시 공개 전 관리자 큐**, **이미지 첨부 시 클라이언트 EXIF·캡처 의심 메타 검증 후 서버 비전 판정**, **서로 다른 신고자 수 누적이 임계치 이상일 때 이용 제한 상태 갱신(기존 상위 제재 우선 유지)**, **운영자의 승인·반려·비공개·제재 해제**를 하나의 안전 처리 흐름으로 묶은 시스템·방법입니다. 지도 UI 측면에서는 **성향 속성 기반 복수 오버레이** 및 **시간 버킷 기반 난수 시드 군중 분포 표시**를 부가 실시예로 둘 수 있습니다.

## 2. 구현으로 고정된 핵심 사실(특허 실시예 정리용)

| 축 | 요약 |
|----|------|
| 텍스트 판정·보류 | `spot_reports.status`에 `held`. Edge `moderate-user-content`. |
| 보류 시 일반 피드 비노출 | 공개 피드 조회가 `status = 'verified'`만 사용 → `held`는 노출되지 않음. |
| 관리자 보류 해제 | RPC `admin_resolve_held_spot_report`: `verify` / `reject` / `hidden`. |

| 다인 신고·임계 | SOS 신고·콘텐츠 신고의 **서로 다른 `reporter_id` 수** 합산, **5명 이상** 시 `user_moderation`을 `suspended_reports`로 갱신(기존 `banned`, `suspended_admin`, `suspended_ai`는 유지). |

| 관리자 예외 | SOS 일일 1회 제한 트리거에서 `is_spotvibe_admin()`이면 통과. |

## 3. 청구 구조(내부 초안)

- **방법 독립항**: 입력 텍스트 수신 → 텍스트 판정·보류 큐 → (선택) 이미지 메타 검증·비전 판정 → 다인 신고 누적·제재 → 운영자 조치.  
- **시스템 독립항**: 동일 블록을 수신부·판정부·검증부 등 장치 명칭으로 서술.  
- **종속항**: 보류 시 피드 차단·검토 후 공개, 시간 창·스크린샷 태그, 신고자 수·제재 우선순위, 지도 복수 오버레이, 시간 시드 군중 좌표.

상세 청구 문구·매핑표는 **같이 보내는** `스팟바이브-변리사-전달-전체본.md`(또는 PDF/HWP)와 `스팟바이브-청구항-초안-요약.md` 를 보시면 됩니다.

## 4. 변리사 측에 요청드리고 싶은 검토 포인트

1. **선행조사 키워드**: 위치 기반 UGC 모더레이션, EXIF·스크린샷 탐지, 비전 남용 판정, 다인 신고 임계 자동 제재, 하이퍼로컬 SOS·비상 신고.  
2. **청구 범위**: 독립항에서 “임계치·상태명·우선순위 집합”을 어느 수준까지 일반화할지.  
3. **분리 출원 여부**: UGC 안전 파이프라인(핵심) vs 지도 오버레이·군중 분포(부가) — 단일 명세 vs 분리 검토.  
4. **실시예 보강**: 시간 창(분)·카테고리 코드·관리자 식별 방식 등은 **코드·마이그레이션과 동일 명칭**으로 명세·도면에 맞출 것을 권장.

## 5. 생활 안전 시설물(화장실·운동기구·소화기·공공 쓰레기통) 특허 가능성 요약

- **현재 구현 근거**: 지도 UI에서 소화기·화장실·야외운동·공공 쓰레기통(분리수거함) 카테고리 레이어/시트/제보 흐름은 확인되며, 소화기 긴급 조회 진입점이 분리되어 있습니다. 다만 일부 데이터 경로는 데모/클라이언트 중심 동작이 포함됩니다.
- **특허 가능성 관점**: 단순 시설물 지도 표시는 선행기술이 많아 약할 수 있으나, 시민 제보 검증(메타/AI), 허위신고 임계 정화, 화재 맥락의 소화기 즉시 탐색 연계를 결합하면 차별 포인트를 구성할 수 있습니다.
- **변리사 전달 문구(요약)**: “화장실·운동기구·소화기·공공 쓰레기통 찾기 기능은 단순 POI 검색이 아니라 검증·정화·긴급상황 연계를 포함한 공익형 위치정보 처리 파이프라인으로 정리 가능하며, 구현 완료 항목과 확장 실시예(백엔드 자동화)는 분리 표기해 청구범위를 조정 요청드립니다.”
