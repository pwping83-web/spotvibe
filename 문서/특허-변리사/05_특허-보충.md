## 1. 본 보충 자료의 위치

- **기존 초안**은 하이퍼로컬 지도·탐색 앵커·이동 프로필 기반 AI UI·OAuth 등 **프런트 중심 흐름**에 무게가 있다.
- 아래 항목은 **서버( Supabase ) 테이블·RLS·SECURITY DEFINER RPC·트리거**와 결합된 **사용자 생성 콘텐츠(UGC)·포인트 경제·조건부 실매칭·집단 신고**에 해당한다.
- 출원 전 **선행기술조사**와 청구항 구성 시, 기존 초안과 **합쳐서 읽거나**, **별도 발명 단위**로 쪼갤지는 변리사와 협의가 필요하다.

## 2. 추가·강화된 기능 요약

### 2.1 프로필·위치 공유 정책(백엔드)

- `profiles`에 **탐색 좌표**(`explore_lat`, `explore_lng`), **위치 모드**, 연령·성별·MBTI·혈액형·활동 태그·성별 인파 선호 등 **매칭·지도 표시용 속성**을 저장한다.
- RLS로 **본인 행은 전체 CRUD**, 타인에 대해서는 **좌표가 설정된 경우에 한해** 위치·필터 일부를 읽을 수 있게 하여, **다른 로그인 사용자 간 실시간 근접 매칭 쿼리**가 가능하다.

### 2.2 현장 제보(Spot Report) 파이프라인

- **사진 URL + 위도·경도 + (선택) 장소명·설명**을 `spot_reports`에 저장한다.
- 제보 시 **지오로케이션 `maximumAge: 0`** 등으로 **“지금 이 자리” 좌표**만 받도록 하여, 캐시된 과거 위치로의 제보 왜곡을 줄인다.
- 스토리지 버킷(`spot-photos`)에 **사용자 ID 폴더 경로**로 업로드·삭제를 제한하고, **공개 읽기**로 피드 노출을 지원한다(MIME 제한 완화 마이그레이션 포함).
- **클라이언트 측 얼굴 영역 모자이크**(Canvas 기반) 후 업로드하는 경로를 두어, **촬영물 UGC와 프라이버시 처리**를 결합한다.

### 2.3 제보 상태·검증·포인트 적립

- 제보 `status`: `pending` / `verified` / `rejected` 및 **피드 비노출용** `hidden`.
- **외부 AI(Edge Function 등) 없이** 본인 `pending` 제보를 `verified`로 올리는 RPC(`autoverify_own_spot_report`)로, 앱이 **단일 진입점**을 갖게 할 수 있다(라벨·카테고리·사유 필드에 고정 문구 부여).
- **별도 관리자 전용 RPC**로 동일 계열 검증을 제한적으로 허용하는 설계(테스트·운영 목적)가 마이그레이션에 존재한다.
- `verified` 확정 시 **DB 트리거**로 `profiles.contribution_points`를 **일정량 가산**하여, **상태 전이와 보상의 원자적 연동**을 서버에서 보장한다.

### 2.4 지도·이벤트 레이어와의 연동(클라이언트 집계)

- 검증된 제보를 조회한 뒤, **고정 반경(예: 200 m) Haversine 클러스터링**과 **시간 창(예: 최근 2시간)**으로 **“현장 이벤트 덩어리”**를 만들고, **2건 이상이면 ‘확인된 이벤트’**로 표시하는 등 **시공간 집계 규칙**이 있다.
- 주기적 재조회(폴링 간격)로 **실시간에 가까운 피드**를 유지한다.

### 2.5 다인 신고·임계값 자동 숨김

- `spot_report_flags`에 **(제보 ID, 신고자 ID)** 유니크 제약으로 **사용자당 1회**만 집계.
- SECURITY DEFINER RPC(`flag_spot_report`)에서 **본인 제보 신고 불가**, **verified만 신고 가능** 등을 검사한 뒤 플래그 삽입.
- **서로 다른 사용자 수가 임계값(예: 20명)**에 도달하면 해당 제보를 `hidden`으로 전환하여 **피드에서 자동 제외**한다.

### 2.6 기여 포인트 소모형 “바이브 부스트”(매칭 세션)

- `profiles`에 **세션 만료 시각**(`peek_session_until`)을 두고, RPC(`start_peek_match_session`)로 **기여 포인트 일정량 차감** 후 **48시간** 동안 세션을 연다.
- 세션이 유효한 동안만 **MBTI·혈액형·활동 태그** 등 프로필 필터가 **타 사용자 실매칭 쿼리**에 반영되고, 비세션 시에는 해당 필터를 **빈 집합으로 두어 매칭에서 사실상 제외**하는 **토글형 매칭 깊이**를 구현한다.
- **JWT 이메일이 사전 정의 관리자와 일치**하는 계정에 대해서는 포인트 차감 없이 세션만 연장하는 **운영·시연용 분기**가 있다(실서비스 시에는 환경설정·역할 테이블로 치환 권장).

### 2.7 근접 사용자 표시와 신선도

- 매칭용 후크는 **반경 박스 필터 + 클라이언트 측 “최근 갱신” 임계(예: 5분)**로 오래된 점을 제거한다.
- DB에는 `last_seen_at` 컬럼·인덱스가 추가되어 **위치 신선도**를 서버에서 필터링할 여지가 있으나, 클라이언트는 호환성을 위해 **`updated_at` 기반**으로 동작하도록 되어 있다.

### 2.8 인증·앱 라우팅 보조

- OAuth 복귀 직후 URL의 code/hash와 세션 교환 타이밍을 고려한 **짧은 지연 후 `authReady`** 처리 등, **세션 레이스 완화** 로직이 있다.

## 3. 특허·IP 검토 시 참고(비법률 메모)

- **단순 “지도에 사진 올리기”**가 아니라, **좌표 신선도 제약 + 스토리지 경로 정책 + 상태 머신 + 트리거 포인트 + RPC 신고 임계값**이 **하나의 UGC 품질·남용 방지 파이프라인**으로 묶여 있으면, **시스템 청구** 관점에서 정리할 여지가 있다(진보성은 선행조사 필수).
- **포인트 적립(검증) ↔ 포인트 소모(세션) ↔ 조건부 필터 반영**은 **게이미피케이션과 LBS 매칭의 결합**으로, **입력·조건·효과**를 수식·플로차트로 고정하면 설명이 쉬워진다.
- **집단 신고에 의한 자동 숨김**은 추천·노출 안전장치로서 **플랫폼 운영 발명** 축으로 검토 가능하나, **임계값·예외(오남용)**는 실시예와 함께 적는 것이 좋다.
- 관리자 예외는 **출원서에는 일반화**(예: “운영자 권한 플래그”)하고, **하드코딩 이메일**은 제품 코드 정리와 별도로 문서에 노출을 최소화하는 편이 안전하다.

## 5. SOS 하이퍼로컬 도움 신호 시스템 — 특허 보충 (2026-05-05 추가)

> 작성 목적: 2026년 5월 추가된 SOS 기능 전체를 발명 포인트 단위로 열거.  
> 기존 초안(현장제보·포인트·매칭 섹션)과 별도로 변리사에게 전달하거나, 별도 출원 단위로 검토 요청할 수 있도록 독립 기술.

---

### 5.1 기능 개요

스팟바이브 SOS 도움 신호는 **공식 응급서비스(119/112)의 대체가 아닌**, 반경 3 km 이내 로그인 이용자에게 도움 상황을 실시간으로 알려 커뮤니티 차원의 추가 지원을 유도하는 **하이퍼로컬 시민 상호 도움 플랫폼**이다.

---

### 5.2 특허 포인트 ④ — 상황 유형 분류 기반 공식 기관 연계 강제 확인형 SOS 발신 시스템

#### 기술적 과제

기존 SOS·긴급 신호 앱은 단일 버튼으로 단순 알림만 발송하거나, 반대로 공식 응급 신고만 연결한다. 커뮤니티 도움 신호와 공식 신고의 역할을 혼동하게 되어 **오남용**(장난 발신, 가벼운 요청에 의한 반복 발신)이나 **과소 신고**(정작 119를 먼저 불러야 하는데 앱만 사용) 문제가 발생한다.

#### 해결 수단 (구체적 구현)

스팟바이브 SOS는 다음과 같은 **다단계 강제 확인 흐름**을 구현한다.

```
[Step 1 — 법적 경고 강제 노출]
  ① 공식 응급서비스 대체 불가 경고문
  ② 119 / 112 단축 다이얼 버튼 (앱 내에서 바로 전화 가능)
  ③ "확인했습니다" 동의 버튼을 눌러야만 Step 2로 진행
     → 버튼은 위치 미취득 시 disabled 처리 (위치 없이 발신 불가)

[Step 2 — 상황 유형 선택]
  > [정정] 아래 8유형은 초기 기획안입니다.
  > 현행 DB/UI는 4유형(fire / public_safety / missing / medical)으로 통합됩니다.
  > 특허 도면/실시예는 4유형 기준으로 작성하세요. (참고: 06항 §6.3, src/types/sos.ts)

  [현행 4유형]
  ① fire         : 화재 목격/대피
  ② public_safety: 치안 위협/범죄 연계 의심 (기존 safety_* 통합)
  ③ missing      : 고립/조난/이동 불가 (기존 tourist_help 통합)
  ④ medical      : 의료 긴급 (기존 medical_* 통합)
  ② 유형별 callFirst 속성: '119' | '112' | null
     → callFirst가 있는 유형은 카드에 "먼저 119 신고" / "먼저 112 신고" 뱃지 표시
  ③ 상황 메모(선택, 최대 120자) 입력

[Step 3 — 발신]
  ① GPS 좌표 + 유형 + 메모 → sos_signals 테이블 INSERT
  ② expires_at = NOW() + INTERVAL '15 minutes' (DB 기본값)
  ③ Supabase Realtime 구독 → 반경 3 km bbox 내 타 이용자에게 실시간 전파
```

유형별 `callFirst` 분기는 의료·화재·위협 상황에서 공식 기관에 먼저 신고할 것을 **UI 수준에서 강제 안내**하며, 법적 경고 단계 없이 즉시 발신하는 것을 구조적으로 방지한다.

#### 청구항 후보 (안)

> **청구항 제4항**: 이용자가 도움 신호를 발신하기 전 공식 응급서비스 비대체 경고문과 직통 전화 버튼을 강제로 노출하고, 상황 유형을 복수의 분류 중 하나로 선택하도록 하되, 각 유형에 매핑된 공식 신고 기관 속성(callFirst)에 따라 공식 기관 신고 선행 안내 뱃지를 해당 유형 카드에 표시함으로써, 커뮤니티 도움 신호와 공식 신고의 역할을 UI 흐름 수준에서 분리·강제하는 방법 및 시스템.

### 5.3 특허 포인트 ⑤ — 만료 시각 연동 실시간 투명도 감쇠 지도 마커 시스템

#### 기술적 과제

실시간 도움 신호를 지도에 표시할 때, 오래된 신호와 최신 신호를 동등하게 표시하면 이미 해결됐을 수도 있는 신호에 이용자가 반응하는 **정보 신선도 혼란**이 발생한다. 그렇다고 만료 시까지 신호를 즉시 삭제하면, 신호의 남은 유효시간을 이용자가 인지할 수 없다.

#### 해결 수단 (구체적 구현)

스팟바이브는 SOS 마커의 **시각적 투명도를 만료까지 남은 시간에 비례해 실시간으로 감쇠**시키는 방식을 구현한다.

```
sosOpacity(expiresAt, totalMinutes = 15):
  left = max(0, round((expiresAt - now) / 60000))   ← 남은 분
  return 0.35 + (left / totalMinutes) × 0.65

  → 발신 직후(15분 남음): opacity = 0.35 + 1.0 × 0.65 = 1.00 (완전 불투명)
  → 7.5분 경과(7.5분 남음): opacity = 0.35 + 0.5 × 0.65 = 0.675 (반투명)
  → 만료 직전(0분 남음):  opacity = 0.35 + 0.0 × 0.65 = 0.35  (최소 투명)
```

이 투명도 값은 클라이언트에서 60초 주기 인터벌로 재계산되어 Leaflet CircleMarker에 실시간 반영된다. 이를 통해 지도 위에서 신호의 **경과 시간·신선도·긴박함**을 별도 UI 없이 시각적으로 직관적으로 표현한다.

#### 청구항 후보 (안)

> **청구항 제5항**: 도움 신호의 만료 예정 시각과 현재 시각의 차이를 설정된 총 유효 시간으로 나눈 비율을 지도 마커의 투명도에 선형 매핑하여, 신호가 오래될수록 마커가 점차 투명해지도록 실시간으로 시각화함으로써, 신호의 신선도를 별도 UI 요소 없이 표현하는 방법.

---

### 5.4 특허 포인트 ⑥ — 활성 SOS 신호 중 진동·푸시 알림 복합 귀환 유도 시스템 (BM 특허 / 방법 특허)

#### 기술적 과제

기존 SOS·긴급 신호 서비스에서는 사용자가 신호를 발신한 뒤 앱을 닫거나 다른 탭으로 이동하면, 신호가 아직 활성 상태임을 인지하지 못해 **해제 없이 방치**되는 경우가 발생한다. 이는 신호를 수신한 이웃이 도착했을 때 이미 상황이 종료됐음에도 신호가 남아 있어, **이웃의 불필요한 이동과 신뢰도 저하**를 야기한다.

#### 해결 수단 (구체적 구현)

스팟바이브는 신호 활성 기간 동안 다음 **복합 귀환 유도 메커니즘**을 동시에 실행한다.

```
[진동 리마인더]
  interval = 45,000ms (45초)
  pattern  = [300ms ON, 150ms OFF, 300ms ON, 150ms OFF, 300ms ON]
  → navigator.vibrate(pattern) 호출
  → SOS 해제 또는 만료 시 clearInterval

[브라우저 푸시 알림]
  interval = 120,000ms (2분)
  title    = "🆘 SpotVibe — SOS 신호 발신 중"
  body     = "신호가 아직 켜져 있어요. 상황이 해결됐다면 앱에서 신호를 해제해 주세요."
  tag      = "sos-active-reminder" (동일 태그로 덮어쓰기 — 알림 누적 방지)
  renotify = true (같은 tag여도 다시 표시)
  → SOS 해제 또는 만료 시 clearInterval

[시각 긴급 애니메이션]
  버튼: 1.1초 주기 빨간·주황 교번 팽창·수축·글로우 (CSS keyframe)
  배너: 지도 최상단 고정, 1.4초 주기 배경색 교번 플래시
  → myActiveSosSignalId 상태 기반 조건부 렌더링
```

최초 활성화 시 진동을 즉시 1회 실행하고, 이후 45초 주기로 반복함으로써 앱을 화면에서 벗어나 있는 사용자에게도 **물리적·시각적·알림 3중 귀환 유도**를 제공한다. 신호 해제 혹은 만료 시 모든 인터벌이 즉시 클리어된다.

#### 청구항 후보 (안)

> **청구항 제6항**: 위치 기반 도움 신호가 활성 상태인 동안, 설정된 주기마다 모바일 디바이스 진동(일정 패턴의 on/off 시퀀스)과 브라우저 푸시 알림(동일 태그 renotify 방식)을 복합적으로 발동하여 발신자에게 신호 해제를 촉구하고, 신호 상태가 해제(resolved) 또는 만료(expired)로 전환되는 순간 모든 주기 작업을 종료하는 방법 및 시스템.

## 7. BM 특허 포인트 ⑦ — 일상 밀집도 탐색 습관 기반 수동 관찰자→능동 구조자 전환 유도 시스템

> 작성 목적: 2026년 5월 확정된 제품 성장 전략("외로움·호기심 DAU 루프 → SOS 문화 전환")을 발명 포인트로 기술.  
> 기존 §5 SOS 기능 특허와 병행 출원 단위로 검토 요청 가능.

---

### 7.1 기술적 과제 (선행기술과의 차이)

기존 SOS·긴급 신호 앱은 **안전·위기 목적으로만 설계**되어 평상시 사용 동기가 없다.
결과적으로 정작 위기 상황에서 앱을 켜고 있는 주변인(잠재적 구조자)이 거의 없는
**"유령 앱 현상"** 이 발생한다.

이를 해결하려면:
1. 안전·위기와 무관한 **일상적 이유**로 앱을 반복 사용하게 만들어야 하고,
2. SOS 신호를 수신했을 때 **심리적 참여 장벽을 최소화**하는 사회적 증거(social proof)를 제공해야 하며,
3. 참여 행동 이후 **자발적 바이럴 확산**이 일어나도록 해야 한다.

---

### 7.2 해결 수단 (3단 전환 메커니즘)

#### A. 일상 DAU 확보 루프 (호기심 중독 설계)

```
[일상 접속 동기]
  ① 군집 밀도 지도: "지금 이 동네 어디에 사람이 많지?" (사회적 외로움·FOMO 자극)
  ② 연령대·MBTI 오버레이: 내가 관심 있는 인구가 어디 있는지 시각화
  ③ AI 말풍선: 30~60초 간격 자동 출현, 탭 유도 (능동 탐색 없이도 앱이 말을 걸어옴)
  ④ PRNG 드리프트: 1분 주기 점 이동 → 매번 조금 다른 지도 → 재방문 관성

→ 결과: 안전 목적 없이도 하루 수회 앱을 켜는 DAU층 형성
         = SOS 수신 대기자(잠재 구조자) 상시 확보
```

#### B. SOS 목격 → 사회적 증거 노출 (심리적 장벽 제거)

```
[SOS 마커 탭 시퀀스]
  ① 지도 위 붉은 SOS 마커 노출 (반경 3km, 투명도 감쇠)
  ② 호기심으로 탭 → SosPeerSignalSheet 열림
  ③ 이전 해결 사례: AI 검증 현장 사진 + "○분 전 이웃이 도움을 줬어요" 텍스트
     - 사진: verify-sos-photo Edge Function 통과 (EXIF + 비전 검증)
     - 사례가 시각적 증거로 작동 → "나도 할 수 있겠다" 자기효능감(self-efficacy) 유발
  ④ [이 이웃 돕기] 버튼 → 수동 관찰자에서 능동 구조자로 전환

→ 핵심 차별점: 단순 알림이 아닌, 이미 해결된 사례의 AI 검증 사진이
               새 SOS 마커와 연결되어 사회적 증거(social proof)로 기능
```

#### C. 참여 후 바이럴 확산 (자발적 문화 형성)

```
[구조 완료 후 시퀀스]
  ① 히어로 인증서 발급: 유형별 디지털 배지 + 구조 횟수 누적 등급
  ② 원터치 SNS 공유: 인스타그램 스토리 / 릴스 직접 공유
  ③ 운영팀 숏폼 제작: 실제 사례자 인터뷰 → 유튜브·인스타 릴스
  ④ 미디어 확산 → "앱이 아니라 문화"로 포지셔닝
  ⑤ 신규 가입자 유입 → DAU 증가 → SOS 수신 이웃 증가 → 해결률 상승 (루프)
```

## 6. AI·EXIF·집단 신고 연동 UGC 거버넌스 (2026-05 반영)

> **목적**: `스팟바이브-특허명세서-초안.md` §3.4·공익 서술과 맞춘 구현 스냅샷. 변리사 전달 시 명세서 초안과 함께 첨부.

### 6.1 요지

- 실시간 **현장 제보·SOS·실시간 사진** 경로에서 **상업 홍보·성인·범죄 연계·허위·화면 재촬영** 등을 **다층으로 억제**.
- **공익**: 이웃 안전·지역 정보 **신뢰** — “올리는 것”보다 **의심 시 차단·보류**를 우선할 수 있는 구조.

### 6.2 계층별 구현(작동 축)

| 순서 | 층 | 내용 |
|------|----|------|
| 1 | 텍스트 LLM | Edge `moderate-user-content`: 현장 제보 `allow`/`held`/`block`, SOS 메모 `allow`/`block`, JWT 검증. |
| 2 | EXIF·스크린샷(클라) | SOS 첨부: 원본 File에서 촬영 시각·Software 태그 검사 후 압축·비전 호출. |
| 3 | 비전 LLM | `verify-sos-photo`: 재촬영·남용 판정, `abuse` 시 `user_moderation` 연동 가능. |
| 4 | 현장 `held` | 텍스트 보류 시 DB에 비공개 검토 상태 저장, 관리자 RPC로 승인·반려·hidden. |
| 5 | 유형별 신고·집단 제한 | `content_abuse_reports` 등 + 서로 다른 신고자 수 임계 시 `user_moderation` 자동 갱신. |
| 6 | 관리자 | held·신고 큐 화면에서 후행 판독. |

### 6.3 명세·도면 정합성

- **§5.2 Step 2**의 8유형 SOS 서술은 초기안 참고용일 수 있음. **현행 UI·DB**는 **4유형**(화재·치안·조난·구급) — 특허 도면·실시예는 **4유형** 기준으로 통일 권장.
- 출원서에는 모델 벤더를 **일반화**(예: “외부 판정 모듈”)하고 실시예에 구현을 묶는 것이 유리함.

## 8. 공공기관 연계 취약계층 위치 공유 및 이웃 근접 알림 시스템 (2026-05-08 추가)

> 작성 목적: 시각장애인 등 취약계층의 안전을 위한 공공기관 승인 기반 위치 공유 및 일반 시민 근접 알림 시스템을 발명 포인트로 기술.

---

### 8.1 기술적 과제

기존 LBS 앱은 일반 사용자 위치만 다루며, 시각장애인·이동약자 등 도움이 필요한 취약계층의 위치를 주변 시민에게 실시간으로 알리는 공익 특화 시스템이 없다.

또한 취약계층을 임의로 등록하면 개인정보 침해 문제가 발생하므로, **공공기관(구청·주민센터) 승인을 거친 당사자 동의 등록** 체계가 필요하다.

기존 SOS 앱은 **당사자가 직접 신호를 보내야** 하지만, 시각장애인의 경우 스마트폰 조작이 어렵거나 위기 인지가 늦을 수 있어 **상시 수동 위치 공유** 방식이 더 효과적이다.

---

### 8.2 해결 수단 (구체적 구현)

`
[공공기관 연계 등록 흐름]

## 9. 사용자 제보 기반 생활 안전 시설물 위치 레이어 시스템 (2026-05-08 추가)

> 작성 목적: 네이버·카카오 지도에 없는 소화기·야외 헬스장·약수터 등을 시민 제보로 지도화하고, 소화기 긴급 조회 기능과 연계하는 시스템을 발명 포인트로 기술.

---

### 9.1 기술적 과제

공공 산책로·산·공원 내에는 **소화기 보관함, 야외 헬스장, 약수터, 안내 시설** 등 생활 안전 시설물이 다수 존재하지만, 상업 지도 서비스에는 등록되지 않아 일반 시민이 위치를 알 수 없다.

특히 소화기의 경우, 초기 화재 시 **주변 소화기 위치를 즉시 파악하지 못해** 재산·인명 피해가 발생하는 문제가 있다. 집 안 소화기가 유효기간 만료 또는 작동 불량인 경우에도 마찬가지다.

기존 크라우드소싱 지도(구글 Maps, 카카오맵 플레이스)는 상업 시설 중심이며, **비상업적 안전 시설물의 실시간 카테고리별 레이어 표시와 긴급 조회 기능이 없다**.

---

### 9.2 해결 수단 (구체적 구현)

`
[시설물 제보 플로우]

  ② 카테고리 선택 (소화기 / 화장실 / 야외 헬스장 / 공공 쓰레기통·분리수거함 / 흡연실·흡연부스 / 약수터 / 기타)
  ③ EXIF GPS 좌표 자동 추출 → 지도 핀 자동 세팅
  ④ AI 텍스트 + 비전 검증 통과 → spot_facilities INSERT
  ⑤ 지도 확대 시 카테고리별 마커 표시
  ⑥ 마커 탭 시 제보 사진·거리·제보 문구가 포함된 상세 시트 표시
  ⑦ 허위 제보 신고 누적 카운트 관리, 5회 도달 시 해당 마커 자동 삭제/비노출

[소화기 긴급 조회 버튼]

  - 탭 시 반경 3km 내 제보 소화기 위치 강조 표시 + 거리순 목록
  - "가장 가까운 소화기: 북쪽 150m" 형태로 즉각 안내
  - SOS 화재 유형 발신 시 소화기 조회 연동 안내

[시간 기반 유효성]

## 10. 시민 제보 기반 동네 숨은 장소 카테고리 마커 및 계절 필터 시스템 (2026-05-08 추가)

> 작성 목적: 상업 지도에 없는 비상업적 생활 장소를 카테고리·날짜·계절 기반으로 지도화하는 시스템을 발명 포인트로 기술.

---

### 10.1 기술적 과제

기존 상업 지도 서비스(네이버 지도, 카카오맵)는 **상업 시설 중심** 데이터베이스이며, 산·공원·골목의 비상업적 생활 장소(무료 야외 헬스장, 꽃길, 배드민턴장, 숨은 쉼터 등)는 등록되지 않는다.

이러한 장소는 **지역 주민만 알고 있으며**, 시즌에 따라 가치가 달라지는 계절성(벚꽃, 철쭉, 단풍 등)이 있다. 기존 앱은 **이런 정보를 날짜 기반으로 카테고리화하여 실시간 공유하는 시스템**이 없다.

`
[카테고리 제보 흐름]

  ② 대분류 카테고리 선택 (꽃·자연 / 운동·체육 / 산책·풍경 /

  ③ EXIF GPS + 촬영 일시 자동 추출
  ④ AI 텍스트 + 비전 검증 통과 → spot_facilities INSERT
