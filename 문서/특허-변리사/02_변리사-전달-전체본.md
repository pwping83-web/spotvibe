## 2. 청구항 초안 전문(방법)

출원 전략상 **방법**과 **시스템**을 분리해 두는 것이 일반적입니다. 번호·용어는 변리사 측에서 재번호·통합할 수 있습니다.

### 2.1 방법 청구항

#### 제1항 (독립항 — 방법)

지리 기반 사용자 생성 콘텐츠를 처리하는 컴퓨터 구현 방법으로서,

(a) 사용자 단말로부터 입력 텍스트를 수신하는 단계;  
(b) 상기 입력 텍스트를 텍스트 판정 모델에 입력하여 허용, 보류, 차단 중 하나로 분류하고, 보류인 경우 공개 전 검토 큐에 등록하는 단계;  
(c) 이미지가 첨부된 경우, 클라이언트에서 촬영 시각 메타데이터 및 캡처 의심 메타데이터를 검증하는 단계;  
(d) 상기 (c) 단계에서 통과한 이미지에 대해 비전 판정 모델로 남용 여부를 판정하는 단계;  
(e) 복수의 상이한 신고자 식별자에 기초한 신고 누적값이 임계치 이상이면 이용자 제한 상태를 갱신하는 단계; 및  
(f) 운영자 단말로부터 상기 보류 항목 또는 신고 항목에 대한 승인, 반려, 비공개 또는 제재 해제 중 적어도 하나의 조치를 수신하여 반영하는 단계;

를 포함하고, 하이퍼로컬 실시간 채널의 콘텐츠 무결성 또는 공익적 안전성 중 적어도 하나를 확보하는 것을 특징으로 하는 방법.

#### 제2항 (종속)

제1항에 있어서, 상기 (b) 단계에서 보류로 분류된 경우 공개 피드에 상기 콘텐츠를 노출하지 않고, 관리자 검토 완료 후에만 공개 상태로 전이시키는 것을 특징으로 하는 방법.

#### 제3항 (종속)

제1항에 있어서, 상기 (c) 단계는 촬영 시각이 사전 정의된 시간 창 내에 있는지 여부 및 스크린샷 생성 소프트웨어 태그 포함 여부를 확인하는 것을 특징으로 하는 방법.

#### 제4항 (종속)

제1항에 있어서, 상기 (e) 단계는 상기 신고 누적값을 서로 다른 신고자 식별자의 수로 산정하고, 상기 임계치 도달 시 기존 제재 상태가 미리 정한 우선순위에서 더 높은 단계이면 해당 상태를 유지하는 것을 특징으로 하는 방법.

#### 제5항 (종속)

제1항에 있어서, 지도 인터페이스에서 사용자 성향 속성 집합을 복수의 지리 오프셋 및 반경에 매핑하여 복수의 오버레이를 동시에 렌더링하는 단계를 더 포함하는 것을 특징으로 하는 방법.

#### 제6항 (종속)

제1항에 있어서, 현재 시각에 대응하는 시간 버킷 값을 난수 시드로 사용하여 클라이언트 간 동기화 가능한 군중 분포 좌표를 생성하고, 연령대별로 구분된 마커로 지도에 표시하는 단계를 더 포함하는 것을 특징으로 하는 방법.

## 3. 청구항 초안 전문(시스템)

#### 제1항 (독립항 — 시스템)

지리 기반 사용자 생성 콘텐츠를 처리하는 정보처리시스템으로서,

텍스트 수신부;  
상기 입력 텍스트를 허용, 보류, 차단으로 분류하고 보류 시 검토 큐에 등록하는 텍스트 판정부;  
이미지 첨부 시 클라이언트 측에서 촬영 시각 메타데이터 및 캡처 의심 메타데이터를 검증하는 메타데이터 검증부;  
상기 메타데이터 검증부의 통과 이미지에 대해 남용 여부를 판정하는 비전 판정부;  
상이한 신고자 식별자에 기초한 신고 누적값이 임계치 이상일 때 이용자 제한 상태를 갱신하는 신고 집계부; 및  
운영자 단말로부터 보류 또는 신고에 대한 승인, 반려, 비공개 또는 제재 해제 중 적어도 하나의 조치를 수신하여 반영하는 운영 조치부;

를 포함하고, 하이퍼로컬 실시간 채널의 콘텐츠 무결성 또는 공익적 안전성 중 적어도 하나를 확보하도록 구성된 것을 특징으로 하는 정보처리시스템.

#### 제2항 내지 제6항 (종속)

방법 청구항 제2항 내지 제6항과 동일한 한정을, 각각 “텍스트 판정부”, “메타데이터 검증부”, “신고 집계부”, “지도 표시부”, “군중 시뮬레이션부” 등 **장치 명칭**으로 치환한 형태로 서술할 수 있습니다. 시스템 종속항 **전문 전체**는 `문서/특허-변리사/04_청구항-초안-요약.md` 에 있으니, 외부 전달 시 그 파일을 PDF 등으로 **함께 첨부**하십시오.

## 8. 생활 안전 시설물(화장실·운동기구·소화기·공공 쓰레기통) 특허 검토 메모

본 항목은 `문서/제품-기획/02_기능-현황-2026-05.md`의 최신 상태를 기준으로, **구현 완료**와 **확장 실시예(로드맵)** 를 분리해 변리사 전달 시 오해를 줄이기 위한 정리입니다.

### 8.1 현재 구현 근거 (적용됨)

- 지도 UI에서 **시설물 카테고리 레이어**(소화기·화장실·야외운동·공공 쓰레기통)를 구분 표시하고, 카테고리별 시트를 통해 제보/조회 흐름을 제공.
- 소화기 관련 **긴급 조회 UX 진입점**을 별도 버튼으로 분리해, 화재 상황에서 빠른 탐색 시나리오를 구성.
- 마커 클릭 시 사진·설명·신고 등 상세 상호작용 구조를 제공(현재 일부는 데모 데이터 기반 동작 포함).

### 8.2 확장 실시예 (보완 필요 / 로드맵)

- `spot_facilities` 전용 테이블, RLS, 신고 임계 트리거 기반 자동 비노출/삭제 등은 **백엔드 확장 과제**로 분리 기재.
- EXIF/GPS 자동 추출, freshness 갱신, 시간 경과 기반 투명도 감쇠의 서버-클라이언트 일체 파이프라인은 **실시예 고도화** 대상으로 기재.
- 출원 문구에서는 “일부 UI·흐름 구현 + 일부 서버 자동화 실시예”를 명확히 구분해 과장 해석 리스크를 줄일 것.

### 8.3 특허 가능성 관점 (실무 요약)

- 단순 “시설물 위치 표시” 자체는 선행기술 밀집 영역이므로 독립 차별성은 약할 수 있음.
- 다만 **시민 제보 검증(텍스트/비전·메타) + 허위신고 임계 정화 + 화재 맥락 소화기 즉시 탐색 연계**를 결합한 운영 파이프라인은 차별 포인트로 주장 가능.
- 권장 전략: 독립항은 일반화하고, 종속항에서 임계 처리·신선도·긴급 연계를 구체화.
