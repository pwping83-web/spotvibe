# 트래픽·남용·DDoS — 실행 가이드

**전제**: “초당 수만 요청”급 **볼륨 공격**은 앱 로직만으로는 한계가 있고, **엣지(CDN/WAF)·호스팅·Supabase 한도**가 1차 방어선입니다.  
이 문서는 **정상 기능을 최대한 살리면서** 할 수 있는 것부터 적습니다.

---

## 1. 지금 당장 (설정 위주, 코드 변경 없음)

- [ ] **정적 호스팅 앞단**에 CDN이 있으면 캐시·기본 DDoS緩和 활성화(제품: Netlify/Vercel/Cloudflare 등 문서 확인).
- [ ] Supabase 대시보드에서 **프로젝트 플랜·API 한도** 확인 — 한도 초과 시 429/5xx 패턴 파악.
- [ ] **지리적 전면 차단**은 신중히: VPN·해외 거주 사용자 오탐.

**기능 영향**: 기본 CDN은 보통 무감. 과한 봇 챌린지는 모바일 웹뷰에서 UX 저하 가능.

---

## 2. 앱·API 레이트리밋 (도입 시 권장 순서)

**낮은 영향 → 높은 효과 순서**

1. **특정 RPC/Edge Function**에만 IP·사용자당 분당 N회 제한 (예: 비용 큰 AI 호출, 업로드 URL 발급).  
2. **로그인·가입** 엔드포인트에 제한 (브루트포스·대량 가계정).  
3. 전역 공격 시 **WAF 레이트리밗**으로 상향.

**구현 메모 (향후 코드 작업 시)**

- Supabase **Edge Function** 앞에서 `X-Forwarded-For` 또는 Supabase가 넘겨주는 클라이언트 식별자 기준으로 **메모리/Redis 카운터**(외부 캐시 필요 시 Upstash 등)를 쓰는 패턴이 흔함.  
- **PostgreSQL만으로 전역 RPS**를 막기 어렵다 → DB는 **비즈니스 한도**(예: SOS 일 1회)에 집중.

**기능 영향**: 같은 Wi-Fi의 많은 사용자가 한 API에 몰리면 걸릴 수 있음 → **엔드포인트 세분화**와 **임계값 완화**로 조정.

---

## 3. IoT·하드웨어 연동 시 (미래 확장)

- 공개 `anon` 키만 단말에 넣지 말 것.  
- **단말별 자격증명** + **짧은 수명 토큰** + **전용 게이트웨이(Edge)**에서 검증 후 DB로.

---

## 4. 모니터링 알림 (이상 징후)

- [ ] 호스팅·Supabase에서 **에러율·p95 지연·요청 수** 알림 설정.  
- [ ] 급증 시 **가장 비싼 쿼리·Realtime 구독 수**를 대시보드에서 확인.

---

## 5. SpotVibe DB에 이미 있는 방어 (중복 설계 방지)

- SOS 관련 **일일 한도 트리거** 등: 동일 목적의 중복 트리거를 여러 개 두면 디버깅이 어려워짐 → 변경 전 마이그레이션 이력 확인.
