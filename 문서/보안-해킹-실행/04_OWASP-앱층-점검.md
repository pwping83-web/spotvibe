# OWASP 관점 — 앱(Vite·React) 층 점검

**목표**: 릴리스 전에 **고위험 항목만 빠르게** 확인하고, **불필요한 CSP 깨짐** 없이 단계적으로 강화합니다.

---

## A. 인젝션 (SQL 등)

- [ ] DB 접근은 **Supabase 클라이언트·RPC 파라미터**만 사용. 사용자 입력을 문자열로 이어 **raw SQL** 만들지 않기.
- [ ] RPC·Edge Function에서 동적 쿼리 시 **파라미터 바인딩** 필수.

**기능 영향**: 규칙만 지키면 일반적으로 무영향.

---

## B. XSS (크로스 사이트 스크립팅)

- [ ] React 기본 이스케이프 유지. **`dangerouslySetInnerHTML`** 사용처가 있으면 입력 출처·sanitize 정책 명시.
- [ ] URL·링크에 `javascript:` 스킴 등 사용자 입력 반영 금지.

**기능 영향**: strict CSP 도입은 지도·외부 스크립트와 충돌할 수 있음 → **호스팅에서 report-only 모드**로 먼저 로그 수집 후 단계 적용 권장.

---

## C. 인증·세션

- [ ] **토큰을 URL 쿼리**에 붙이지 않기(리퍼러 유출).  
- [ ] 로그아웃·계정 전환 시 **클라이언트 상태·스토리지** 정리.

**기능 영향**: 없음(권장 사항).

---

## D. 민감 데이터 노출

- [ ] 프로덕션 빌드에서 **source map 공개** 여부 검토(공개 시 저장소·키 노출 위험).
- [ ] 브라우저 콘솔·Sentry 등에 **JWT·개인정보 전체** 남기지 않기.

---

## E. 의존성 취약점

```bash
npm audit
```

- [ ] **high/critical**은 릴리스 전 검토. 자동 major 업그레이드는 회귀 테스트 후.

---

## F. 헤더 보강 (프로덕션 호스팅 설정)

호스팅(Netlify/Vercel/CloudFront 등)에서 설정 가능한 경우:

- [ ] **HTTPS 강제** (HSTS)  
- [ ] **X-Content-Type-Options: nosniff**  
- [ ] **Referrer-Policy** (필요 수준만)

CSP는 Leaflet·외부 타일·카카오 등 연동 시 **깨짐 테스트** 후 적용.
