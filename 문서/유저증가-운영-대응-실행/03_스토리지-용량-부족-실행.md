# 스토리지(사진) 용량 부족 — 실행 가이드

SpotVibe에서 사진은 주로 Supabase **Storage** 버킷에 올라갑니다. 마이그레이션 기준 버킷 이름은 아래와 같습니다.

| 버킷 ID | 용도 | 참고 마이그레이션 |
|---------|------|-------------------|
| `spot-photos` | 현장 제보 사진 | `file_size_limit` 15MB 등 (`20260502100002_spot_photos_mime_and_limits.sql`) |
| `sos-photos` | SOS 현장 사진 | 단건 약 10MB 제한 (`20260508120000_sos_signals_photo_and_storage.sql`) |

---

## 1. 지금 당장 (대시보드)

- [ ] Supabase Dashboard → **Storage** → 버킷별 **용량·파일 수** 확인.  
- [ ] 플랜별 **Storage 총량 한도**가 얼마인지 Billing/문서에서 확인.

---

## 2. 업로드 실패 메시지별

| 사용자에게 보이는 느낌 | 확인 |
|--------------------------|------|
| 파일 너무 큼 | 버킷 `file_size_limit`, 클라이언트 압축·리사이즈 |
| 413 / Payload | 위와 동일 + 리버스 프록시 제한(호스팅이 API 경유 시) |
| 할당량 초과 | Supabase **플랜 업그레이드** 또는 오래된 파일 정리 |

---

## 3. 운영 정책(코드·배치 작업) — 미리 결정해 둘 것

**정리(Lifecycle) 없이 무한 저장은 플랜 비용만 키웁니다.**

- [ ] **보존 기간**: 예) 제보·SOS 사진 URL이 DB에 남아 있어도, Storage 객체는 N일 후 삭제할지 정책 결정.  
- [ ] **삭제 시 DB**: `photo_url`을 비우거나 “만료됨” 표시할지 (제품 정책과 맞출 것).  
- [ ] **법·개인정보**: 보존·삭제는 개인정보 처리방침과 일치해야 함.

실행 방식 예 (팀이 선택):

1. **수동**: 관리자 스크립트로 오래된 `storage.objects` + DB 컬럼 정리.  
2. **자동**: Edge Function + Cron(스케줄)으로 주기 삭제 (구현 시 별도 개발).

이 저장소에는 **자동 만료 삭제 배치가 필수로 들어가 있지 않을 수 있음** → 트래픽 붙기 전에 정책만이라도 문서화 권장.

---

## 4. 단기 완화(비용 vs 효과)

- [ ] 클라이언트에서 **업로드 전 리사이즈**(긴 변 1920px 이하 등)로 평균 용량 절감.  
- [ ] **HEIF/고해상도** 허용은 유지하되, 안내 문구로 “용량 큰 사진은 실패할 수 있음” 고지.  
- [ ] 플랜 업그레이드가 가장 빠른 해결인 경우가 많음 → `04_Supabase-플랜-한도-확장.md`.

---

## 5. 장애 시 사용자 메시지

- “일시적으로 사진 저장이 지연됩니다” 등 **짧은 안내** + 재시도.  
- 원인이 **용량 한도**면 복구 예상 시각이 있을 때만 공지.
