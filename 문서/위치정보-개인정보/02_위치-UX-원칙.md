# 위치·프라이버시·UX — 다른 프로젝트/Cursor 핸드오프

## 세 줄 방향 (다른 Cursor에 그대로 붙여 넣기)

1. **타인에게는 행정구역만** — 시·구(또는 정책으로 정한 단계)까지만 공개하고, 정밀 좌표는 비공개.  
2. **근접도는 구·동 단위(또는 격자)** — 상대 GPS를 UI에 찍지 않고 행정단위·geohash 등으로만 근사.  
3. **카드·피드는 상대적 동네 라벨** — 「우리 동네 / OO구」처럼 내 기준 프레이밍을 우선.

---

## 1. 위치 정밀도를 나 / 상대로 나눈다 (핵심)

- **나(본인):** GPS·지도 핀·시·구·동·좌표까지 써서 맞춤·거리 계산의 기준점을 잡을 수 있음 (예: `UserLocationContext`, `LocationPickerModal`, 역지오코딩 등).
- **상대(타인):** 공개 UI에는 **시·구**(`region_si`, `region_gu`) 수준만 노출.
  - 예: `MemberProfilePage`에서 `region_si` + `region_gu`만 이어서 한 줄로 표시 (`MapPin` + 텍스트).
- 상대의 정확한 `lat`/`lng`를 프로필·카드에 찍어 주지 않음 → 스토킹·분쟁 리스크를 줄임.

**전달 문장:**  
「타인에게는 **행정구역 단위(시·구)**까지만 보여 주고, 좌표는 내 기준 + 서버/목록 로직에만 쓴다.」

---

## 2. “가까움”은 구(district) 단위 거리로 표현한다

- `DogSittersPage` 등: **내 구(및 추가 관심 구)**를 기준으로, 상대 구 이름끼리 거리를 계산해 “몇 km” 느낌을 줌 (`calculateDistance`, `referenceDistricts`).
- 즉 실시간 상대 GPS가 아니라 **구 단위 대표점 / 행정단위 매핑**으로 근접도를 만든다.

**전달 문장:**  
「근접도는 상대 좌표 노출 없이, **행정구역(구) 간 거리**로만 표현한다.」

---

## 3. 목록 카드에서는 ‘우리 동네 vs 타 구’ 같은 상대 프레이밍

- `MeetupCard`: 내 `userDistrict`와 글의 `meetup.district`를 비교해  
  - 같으면 **「우리 동네」** 배지,  
  - 다르면 상대 구 이름만 작은 칩으로 표시.

**전달 문장:**  
「절대 좌표 대신 **내 동네와 같/다름**으로 감정·정보 밀도를 조절한다.」

---

## 4. 저장·검색 단위를 문자열로 통일한다

- DB/프로필에 `region_si`, `region_gu` 등 **고정 필드**로 두고, 표시는 `join` / `formatRegionInline` 같은 **한 가지 규칙**으로 맞춤.
- 역지오코딩 후 내부 행정명 사전과 매칭해 표기 흔들림을 줄임 (`matchKakaoToRegion` 등).

**전달 문장:**  
「위치는 좌표만 쌓지 말고, UI·필터·거리에 쓸 **행정단위 문자열을 1차 데이터**로 둔다.」

---

## 5. 다른 프로젝트에 복사할 체크리스트

- [ ] 타인 공개: 시·구(또는 동까지) 중 **어디까지 허용할지 정책으로 박고**, 그 이상은 비공개.
- [ ] 거리/근처: 가능하면 **구/동 단위 중심점** 또는 **격자(geohash)**로 근사.
- [ ] 카드/피드: “내 기준 상대적 라벨”(우리 동네 / OO구) 우선.
- [ ] 내 위치: GPS는 **동의·토글·실패 시 수동 선택** UX와 짝.
- [ ] 법·정책: 만남·연결 서비스면 위치 **최소 수집·목적 외 비사용**을 개인정보처리방침과 맞출 것.

## 코드 레퍼런스 (다른 저장소에서 찾을 키워드)

| 개념 | 예시 (타 저장소) |
|------|-------------------|
| 타인 프로필 지역 한 줄 | `MemberProfilePage` — `region_si`, `region_gu`, `regionLine` |
| 내 기준 vs 글의 구 | `MeetupCard` — `userDistrict`, `meetup.district`, 우리 동네 |
| 구 기준 거리·기준 구 목록 | `DogSittersPage` — `referenceDistricts`, `distForDistrict`, `calculateDistance` |
| 내 위치 스냅·저장 | `UserLocationContext`, `LocationPickerModal`, `KakaoMapPicker` |

## SpotVibe(본 저장소) 매핑 — 같은 철학을 코드로 볼 때

SpotVibe에는 위 표의 페이지명이 그대로 있지 않을 수 있음. 대응 관계만 적어 둠.

| 원칙 | SpotVibe에서 볼 곳 |
|------|---------------------|
| 내 GPS·DB 반영·토글과 짝 | `src/hooks/useLocationTracking.ts`, `src/app/App.tsx` (`mapLocationShareActive`, `useProfileSync`의 `locationMode` / explore 좌표 분리) |
| 지도 핀·내 위치 UX | `src/app/components/MapArea.tsx` — `MePositionLayers`, 실시간 추적, 탐색 중심과의 겹침 방지 |
| 타인 좌표는 목록/프로필에 정밀 노출하지 않음 쪽 | `src/hooks/useMatchedUsers.ts` — **id 기반 고정 지터**로 지도 점만 근사 표시(매번 랜덤 점프 방지) |
| “지역” 문자열·탐색 단위 | `src/app/constants/exploreRegions.ts`, `ExploreRegionFlowPanel`, `MapArea` 지역 검색·프리셋 |
| 정책·고지 문구 | `PrivacyPage`, `ServiceIntroPage` — 심사/고지용 서비스 설명 |

**요약:** SpotVibe는 “상대 프로필 카드에 시·구 한 줄” 패턴이 아직 다른 앱만큼 분리되어 있지 않을 수 있으나, **지도 타인 점 = 지터된 근사 좌표 + 주기적 서버 조회**, **내 위치 = GPS/토글/DB 분리**로 같은 축을 일부 이미 반영함. 행정구 전용 필드(`region_si`/`region_gu`)를 도입할 때는 위 **§4 문자열 1차 데이터** 규칙을 따르면 됨.
